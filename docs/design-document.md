# ヒト vs AI 反射神経バトル

## 1. ゲームの目的と狙い

### 1.1 概要

**ヒトとAIが、同一のターゲットを追いかけ、より多く補足した方が勝ち**というWebブラウザゲームである。描画領域内に出現するターゲット（円）に、ヒトはマウスを重ね、AIは自機カーソルを重ねることで得点し、制限時間内の合計得点を競う。

### 1.2 教育的狙い

- **強化学習の体感**: ゲームを繰り返すとAIのスコアが伸び、「学習」が目に見える形で伝わる。
- **成長曲線の体験**: 当初はヒトが楽勝 → 次第にAIが迫る → やがてヒトが勝てなくなる、という「AIに追い抜かれる焦燥感」を意図した設計とする。
- **ライトな環境**: ブラウザのみで完結し、ファイルを開くか簡易サーバで配信すればすぐプレイ可能。大規模なバックエンドは想定しない。

### 1.3 注意点（設計上の前提）

- 強化学習を学ぶコンテンツが目的だが、ゲーム自体もヒトが楽しめるUIとする。
- スコア履歴・学習データのリセットなど、繰り返しプレイを前提とした機能を用意する。

## 2. コンセプトと思考

### 2.1 ヒトとAIの「情報の非対称」

- **ヒト**: 目でターゲットの位置を見て、マウスで直接その座標に向かって移動できる。距離や方向を直感的に把握している。
- **AI**: 視覚を持たない。座標（2次元のxy）をそのまま渡すと「位置が分かっている」ことになり、強化学習の題材として弱い。そこで**AIには座標ではなく、1次元の「方向スコア」という形で間接的な情報だけを渡す**という制約を設ける。

### 2.2 方向スコアの考え方

- 各方向に「1pxだけ動いたとき」の**ターゲットとの距離の変化量 Δ**は、三平方の定理で計算できる。
- この Δ をそのまま渡すと、値の大小から「どの方向が近いか」が数学的に復元しやすく、実質的に座標情報に近くなる。
- そこで **Δ を非自明な1次元スカラーに変換**し、「ピクセルをこう変換した」と簡単には推測できない形にする。方向ごとに値の傾向は変わるが、値だけ見ても座標を直接は復元しづらい変換とする。

### 2.3 方向と移動量の役割分担

- **方向（どの方向に動くか）**: 8方向の方向スコアの大小から**計算で一意に決められる**（スコアが最小の方向＝一番近づく方向）。学習対象にしない。
- **移動量（その方向に何px動くか）**: ターゲットまでの距離は方向スコアから直接は分からないが、「8方向スコアのばらつき具合」は距離に依存する（遠いと差が緩やか、近いと如実に差が出る）。この**ばらつきを手がかりに、移動量だけを強化学習で学習**する。遠いときは大胆に、近いときは慎重に、という振る舞いを獲得させる。

### 2.4 成長感の設計

- 未学習時は**1px（最小移動量）が最良**とみなすよう、Q値の初期値を「小さい行動ほど高く」設定する。
- 探索率（ε）も初手から高くしすぎず、探索時は「小さい移動量」を選びやすくする重み付きランダムとする。これにより第1ゲームからノロノロした動きになり、試行を重ねるうちにばらつきの傾向から大胆な移動の価値を学習し、成長が実感できるようにする。

## 3. ゲームシステム

### 3.1 共通要素

- **フィールド**: 800×500 px の矩形。端から20pxはパディングとし、ターゲットはその内側にのみ出現する（直径20pxの円が領域内に必ず収まる）。
- **ターゲット**: 直径20pxの円。誰かが獲得すると、パディング内のランダムな位置に再出現する。
- **ヒト**: マウスカーソル位置で「触れた」とみなす。カーソルはゲームエリア内でも表示し、位置を追いやすくする。
- **AI**: 直径8pxの自機カーソルを持つ。視覚はなく、方向スコアのみを入力とする。

### 3.2 対戦モード（vs）

- **形式**: 同一フィールド上で、1つのターゲットをヒトとAIが**同時に**追うリアルタイム対戦。
- **得点**: ターゲットに**先に触れた方**が1点。ヒトはマウスオーバー、AIは自機がターゲットと重なった時点で獲得。同一フレームで両方触れた場合はヒトを優先。
- **制限時間**: 60秒。経過後に得点の多い方が勝ち。
- **スコア履歴**: 対戦ごとのヒト・AIの得点と勝者をLocalStorageに保存し、一覧表示する（最大30件、履歴はリセット対象外）。

### 3.3 学習モード（learning）

- **目的**: ヒトは参加せず、AIだけがターゲットを追い続け、経験値を積む。放置してAIの動きの改善を眺める用途。
- **ルール**: 制限時間なし。ストップボタンを押すまで延々と続く。ターゲットの当たり判定は**AIのみ**（マウスで触れても得点・リスポーンしない）。獲得数と経過時間はストップ時に確定表示。
- **UI**: タブで「学習モード」を選択。スコア表示は「観戦」「—」、タイマーは「○:○○ 経過」とし、ストップボタンを表示する。

### 3.4 モード切替

- **対戦モード** / **学習モード** をタブ（またはトグル）で切り替え、選択中が視覚的に分かるようにする。
- プレイ中にモードを切り替えた場合は、その時点でゲーム／学習を停止する。

## 4. AIの仕様

### 4.1 方向スコアの定義と変換

- **入力**: AIの現在位置とターゲット位置（ゲーム側のみが保持。AIには渡さない）。
- **計算（2乗距離の変化）**: 8方向それぞれについて、「その方向に1pxだけ仮移動したときの**2乗距離の変化 Δ² = 移動後の dist² − 移動前の dist²**」を算出。1px の「距離変化 Δ」は常に ±1 付近で距離に依存せずばらつかないため、**Δ² を使う**と値の大きさが距離に比例し、遠いときは8スコアのレンジが大きくなり、近いときはレンジが小さくなる。
- **変換（復元困難な1次元スカラー）**:
  `obfuscate(Δ²) = sign(Δ²) × log(1 + |Δ²|)`（CONFIG.DEVIATION_USE_SQ が true のとき。DEVIATION_LOG_SCALE でスケール可能）。Δ² が負なら近づいているので変換後も負のスコアとなり、**スコアが小さいほど「近づく方向」**となる。従来の「距離変化 Δ」と `sign(Δ)×√(|Δ|×10)` も DEVIATION_USE_SQ=false で利用可能。

### 4.2 方向の決定

- 8方向の方向スコアのうち、**最小値**を取る方向を「今のターゲットに一番近づく方向」として採用。純粋な計算で決定し、学習対象ではない。

### 4.3 移動量の候補

- フィールドサイズに応じて**慎重〜大胆**を等比数列で配分する。最小1px、最大は短辺の20%（例: 500pxなら100px）。段階数は13程度とし、重複を除いて昇順に並べた配列を CONFIG.MOVE_AMOUNTS とする。これによりフィールドが変わっても天井がスケールし、追い抜き・行き過ぎの振動を抑えやすい。

### 4.4 強化学習で学習するもの

- **状態 (s)**: 8方向スコアの「ばらつき」。2乗距離を用いた方向スコア時は**8スコアのレンジ（range）**が距離に依存するため、固定範囲で線形写像（SPREAD_RAW_MIN/MAX）し、5区間に離散化（STATE_BINS=5）。従来の距離変化を用いた方向スコア時は観測した raw の min/max で [0,1] に引き伸ばす方式も利用可能。
- **行動 (a)**: 移動量のインデックス（0 〜 MOVE_AMOUNTS.length−1）。各インデックスは上記の移動量候補に対応。
- **報酬**: 1ステップあたり「移動前距離 − 移動後距離」から**ステップ罰（STEP_PENALTY, 例: 0.3）を引く**（近づけば正、追い越せば負。歩くたびに少し損をするので、少ないステップで到達するほど有利）。ターゲットを捕捉したときはボーナス（例: 50 − STEP_PENALTY）を付与。これにより「遠いときは大胆に少ないステップで詰める」メリットが学習されやすくなる。ステップ罰は小さめにし、初期の慎重さや「繰り返すと強くなる」体験が崩れないようにする。
- **アルゴリズム**: 表形式のQ学習。Q(s,a) を更新し、ε-greedy で行動選択。学習率・割引率・εの最小値・減衰は CONFIG.RL で指定。

### 4.5 初期方針（成長感）

- **Qの初期値**: 未訪問の状態では、行動インデックスが小さい（＝1pxに近い）ほどQを高く設定（INITIAL_CAUTIOUS_BIAS）。これにより未学習時は貪欲に選ぶと常に最小移動量となり、ノロノロした動きになる。
- **探索率**: 初期 ε を低め（例: 0.05）にし、探索時も「小さい移動量」が選ばれやすい重み付きランダムとする。学習が進むにつれて ε は減衰し、学習した方策を選ぶ割合が増える。

### 4.6 AIの行動タイミング

- 対戦・学習とも、AIは一定間隔（例: 80ms）ごとに1回だけ「8方向スコア取得 → 方向決定 → 移動量選択 → 移動 → 報酬でQ更新」を行う。フレームごとには動かさず、ヒトとの公平性と観察しやすさのバランスを取る。

## 5. 技術・実装仕様

### 5.1 技術選定

- **フロント**: HTML5 + CSS3 + Vanilla JavaScript（フレームワークなし）。
- **永続化**: 学習データ（Qテーブル・ε）は LocalStorage に保存（CONFIG.RL.STORAGE_KEY）。スコア履歴は別キーで最大件数制限付き。
- **起動**: index.html をブラウザで開く、または npx serve / python -m http.server 等で配信。Docker は必須ではない。

### 5.2 ファイル構成

| ファイル      | 役割                                                                                                                  |
| ------------- | --------------------------------------------------------------------------------------------------------------------- |
| index.html    | 画面構成（モードタブ、スコア、タイマー、フィールド、スタート／ストップ／学習リセット、履歴）                          |
| css/style.css | レイアウト・テーマ（ダークベース）、タブ・学習モード用表示                                                            |
| js/config.js  | 定数（フィールド寸法、ターゲット/自機半径、制限時間、8方向、方向スコア用スケール、RLパラメータ）、MOVE_AMOUNTS の算出 |
| js/rl.js      | 強化学習（状態離散化、Qの初期値・取得・更新、ε-greedy、探索の重み付きランダム、保存・読み込み・リセット）             |
| js/ai.js      | AI（方向スコアから最良方向、ばらつきから移動量選択、RL と連携）                                                       |
| js/game.js    | ゲームループ、ターゲット出現・当たり判定、方向スコア計算（obfuscate）、モード（vs/learning）、経過時間・残り時間      |
| js/app.js     | UI連携（モード切替、スコア・タイマー更新、履歴表示、スタート／ストップ、学習リセット）                                |

### 5.3 主要定数（CONFIG）

- フィールド: 800×500。ターゲット半径 10、AI自機半径 4。制限時間 60秒。
- 方向スコア: DEVIATION_SCALE = 10（従来の距離変化版）、DEVIATION_USE_SQ / DEVIATION_LOG_SCALE（2乗距離版）。
- RL: LEARNING_RATE 0.1、DISCOUNT 0.95、EPSILON_INITIAL 0.05、EPSILON_MIN 0.05、EPSILON_DECAY 0.998、STATE_BINS 5、INITIAL_CAUTIOUS_BIAS 8、STEP_PENALTY 0.3。
- ゲーム: AI行動間隔 80ms。ターゲット出現パディング 20px。

### 5.4 データフロー（対戦モード）

1. スタート → 残り時間 60秒、スコア 0、ターゲット・AI初期位置設定。
2. ゲームループ（requestAnimationFrame）: 経過で残り時間を減算。ヒトのマウス重なりを先に判定 → 得点・リスポーン。続けてAIの重なりを判定 → 得点・リスポーン・報酬でQ更新。どちらにも当たらない場合、80ms経過ごとに方向スコア計算 → AI.getAction → 移動 → 報酬計算・Q更新。ループ終了条件は「残り時間≤0」で、その時点で onGameEnd（履歴追加など）。
3. 学習モードでは残り時間の減算とヒトの当たり判定をスキップし、経過時間のみ加算。終了はストップボタンのみ。

## 6. UI・UX

- **モードタブ**: 対戦モード／学習モードを明確に切り替え、アクティブなタブが分かるスタイル。
- **スコア・タイマー**: 対戦時は「ヒト / 残り○秒 / AI」、学習時は「観戦 — / ○:○○経過 / AI」。
- **スタート・ストップ**: 対戦時はスタートのみ（時間で自動終了）。学習時はスタートで開始、ストップで停止。
- **学習をリセット**: Q と ε を初期状態に戻し、LocalStorage の学習データを上書き。スコア履歴はリセットしない。
- **スコア履歴**: 直近の対戦結果を「日時 — ヒト ○ : ○ AI（勝者）」の形式で表示（最大15件表示、保存は30件まで）。
- **アクセシビリティ**: モード切替に role="tablist" / role="tab" / aria-selected を使用。

## 7. エクスポートJSONの見方

学習データをエクスポートしたJSONは、次のような構造になっている。

### トップレベル

| キー                         | 意味                                                                                     |
| ---------------------------- | ---------------------------------------------------------------------------------------- |
| **Q**                        | 強化学習のQテーブル（後述）                                                              |
| **epsilon**                  | 探索率（0〜1）。0.05 なら 5% の確率でランダム行動、95% で「今のQで最良」の行動を選択する |
| **exportedAt**               | エクスポートした日時（ISO 8601）                                                         |
| **learningModeSessions**     | 学習モードで「ストップ」した回数（試行回数）                                             |
| **learningModeTotalSeconds** | 学習モードの経過時間の累計（秒）。インポート後の表示はここから「続き」になる             |

### Q の中身

- **キー（"0" 〜 "4"）**: **状態**を表す。8方向の方向スコアの「ばらつき」を 0〜1 に正規化し、5区間に離散化したインデックス。
  - **"0"**: ばらつきが小さい → ターゲットから**遠い**と解釈されやすい状態
  - **"4"**: ばらつきが大きい → ターゲットに**近い**と解釈されやすい状態
- **値（配列）**: その状態で「どの移動量を選ぶか」の**価値（Q値）**。配列の**インデックス**が行動（移動量の番号）、**要素の数値**がその行動の評価。
  - インデックス **0** = 最小の移動量（1px に近い・慎重）
  - インデックスが**大きい**ほど = 大きい移動量（大胆）
  - **数値が大きい**ほど、その状態では「その行動が良い」と学習されている。通常は「最大値のインデックス」が貪欲選択で選ばれる。

例: `"4": [210, 200, ..., 254, ..., 200]` なら、状態 4（近い側）では 9 番目の行動の Q が 254 で最大なので、この状態では 9 番目の移動量がもっとも選ばれやすい。

## 8. まとめ

本ゲームは、**視覚を持つヒトと、方向スコアという1次元情報だけを持つAI**が、同じルールでターゲット獲得数を競う。AIは「方向」を計算で決め、「移動量」だけを強化学習で学習する。初期は慎重に動くように設計し、試行を重ねることでばらつきの傾向から遠近を推測し、遠いときは大胆に・近いときは慎重に動くようになる。対戦モードと学習モードをタブで切り替えられ、学習モードではヒトは参加せずAIの成長を観察できる。技術的にはブラウザのみで完結し、学習データは LocalStorage に保存するライトな構成とする。
